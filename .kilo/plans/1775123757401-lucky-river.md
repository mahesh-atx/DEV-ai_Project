1. A separate `.txt` prompt file (LLM-facing description)
2. A standalone tool implementation module
3. A central registry that assembles tools per role

## Phase 2: Create Tool Registry

### `tools/index.js`

A central registry that:
- Loads all tool definitions
- Filters tools by role (like DevAI's current `getToolsForRole`)
- Returns tool schema + prompt for the LLM

```javascript
// Pseudo-structure:
export const TOOLS = {
  bash: { schema: ..., prompt: 'bash.txt', role: ['general', 'debugger', 'orchestrator'] },
  read: { schema: ..., prompt: 'read.txt', role: ['general', 'explorer', 'coder', 'debugger', 'orchestrator', 'plan'] },
  // ... all 25 tools
};

export function getToolsForRole(role) {
  return Object.entries(TOOLS)
    .filter(([_, t]) => t.role.includes(role))
    .map(([id, t]) => ({
      type: 'function',
      function: {
        name: id,
        description: loadPrompt(id),
        parameters: t.schema
      }
    }));
}
```

## Phase 5: Implement New Tools

### 5.1 `multiedit` Tool
- Wraps existing `edit_file` logic
- Accepts array of edit operations
- Applies sequentially with atomic rollback

### 5.2 `apply_patch` Tool
- Parses unified diff format
- Validates hunks before applying
- Supports Add/Delete/Update file operations

### 5.3 `batch` Tool
- Accepts array of tool calls
- Executes in parallel via `Promise.all`
- Returns per-tool success/failure status
- Max 25 tools per batch

### 5.4 `codesearch` Tool
- Calls Exa Code API (already available via `utils/webTools.js`)
- Returns API/SDK/library documentation context

### 5.5 `todowrite` / `todoread` Tools
- Store todos in `.kilo/todos.json` per session
- `todowrite`: create/update/delete todo items
- `todoread`: return current todo list

### 5.6 `codebase_search` Tool
- Semantic/AI-powered code search
- Uses natural language queries
- Returns relevant code spans

### 5.7 `skill` Tool (Optional)
- Load specialized skill definitions from `.kilo/skills/`
- Inject skill instructions into context
- List available skills dynamically

### 5.8 `lsp` Tool (Optional)
- Language Server Protocol integration
- goToDefinition, findReferences, hover, etc.
- Requires LSP server configuration

## Phase 5.9: Interleaved Text + Tool Output

### Requirement
Before executing any tool, the AI must output 2-3 short sentences explaining what it is about to do so the user can follow its thought process. This supports interleaved conversation where text appears before tool calls.

### Implementation

**Add to `prompts/base.txt`** (or the system prompt layer):
```
Before calling any tool, always output 2-3 short sentences explaining what you are about to do.
This lets the user follow your thought process. Do not call tools silently — narrate your actions.
```

**Add to each tool's `.txt` prompt file** (as a footer):
```
IMPORTANT: Before calling this tool, always explain in 2-3 short sentences what you are about to do.
This helps the user follow your reasoning. Never call tools without narrating your intent first.
```

**Agent controller changes** (`engine/agentController.js`):
- The existing loop already processes assistant text messages before tool calls
- No structural changes needed — the prompt instruction is sufficient to drive the behavior
- Ensure the reporter outputs assistant text in real-time (already supported via `emitReporter`)

**Why this works**: The OpenAI API supports interleaved text + tool calls in the same response. The model will output text content first, then tool_calls. The existing message handling already processes both — we just need to instruct the model to do it.

## Risks & Considerations

1. **Tool name changes** — Renaming `run_command` → `bash`, `read_file` → `read`, etc. could break existing sessions or configs. Consider keeping old names as aliases.
2. **LSP tool** — Requires language server setup. Mark as optional/experimental.
3. **codebase_search** — Requires Morph API key. Mark as optional.
4. **batch tool** — Needs careful error handling for partial failures.
5. **apply_patch** — Needs robust diff parser. Consider using existing `diff` npm package.
6. **Backward compatibility** — Existing `prompts/*.txt` files should continue working alongside the new system.
