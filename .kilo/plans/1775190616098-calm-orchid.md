# Strengthening Agent Mode — Improvement Plan

## Overview

After thorough analysis of the RootX CLI agent mode implementation, I've identified **12 concrete improvements** across 4 priority tiers. These leverage existing features and infrastructure — no new dependencies required.

---

## CRITICAL Priority (Fix broken or risky behavior)

### 1. Fix Contradictory Prompt Instructions
**Files:** `prompts/base.txt`, `prompts/soul.txt`

**Problem:** `base.txt` (line 25) demands "RETURN VALID JSON ONLY, NO preamble" while `soul.txt` (line 11) demands "always output 2-3 short sentences explaining what you are about to do." These directly contradict each other and confuse the model.

**Fix:**
- Remove the legacy JSON-only output format section from `base.txt` — the system uses OpenAI-style tool calling, not JSON output
- Remove the legacy `JSON STRUCTURES` block (lines 30-77 in `base.txt`) that describes an unused output pattern
- Change `soul.txt` narration rule to: "Narrate when switching strategies or doing non-obvious steps. Skip narration for routine reads and searches."

**Impact:** Eliminates model confusion, reduces wasted tokens on every turn.

---

### 2. Prevent Wrong Code Replacement in Fuzzy Matching
**File:** `engine/patchEngine.js` (line 147)

**Problem:** The fuzzy matching threshold of `0.8` similarity is dangerously permissive. In files with repetitive patterns (multiple similar functions), an 80% match could replace the wrong code section silently.

**Fix:**
- Raise threshold to `0.92` for multi-line blocks, `0.95` for single-line blocks
- When fuzzy matching fires, include a warning in the tool result: `"Edited file.js (fuzzy match at 87% similarity). Matched: 'function handleX...'"`
- This lets the model self-correct if the wrong section was replaced

**Impact:** Prevents silent corruption of code files.

---

### 3. Implement Message Compaction to Prevent Context Overflow
**File:** `engine/agentController.js` (line 864+)

**Problem:** The `agentMessages` array grows unboundedly. Each tool call adds both the call and result to history. For long-running tasks, this exceeds the model's context window.

**Fix:**
- Import and use `estimateTokens()` from `utils/budgeting.js` at the top of each loop iteration
- When token usage exceeds 75% of the model's context limit, summarize older tool exchanges while preserving:
  - The system prompt (always)
  - The last 2 message pairs verbatim
- Use the existing `prompts/compaction.txt` prompt (currently unused) for summarization
- Track actual token usage from API response `usage` fields

**Impact:** Prevents context window crashes on long tasks, enables much longer agent sessions.

---

## HIGH Priority (Significant capability improvements)

### 4. Add Edit Verification After Each Change
**File:** `engine/patchEngine.js` (line 222)

**Problem:** After an edit is applied, the system says "Successfully edited" without verifying the change actually took effect. If the search string matched unexpectedly, the wrong content gets replaced silently.

**Fix:**
- After each edit, read back the affected lines from the file
- Confirm the replacement text is present in the result
- If not found, report: `"Edit verification failed. Expected 'newText' at line X but found 'actualText' instead."`
- This gives the model concrete feedback to self-correct

**Impact:** Catches edit failures immediately instead of accumulating broken changes.

---

### 5. Enable Parallel Batch Execution
**File:** `engine/agentController.js` (line 1213)

**Problem:** The `batch` tool executes tools sequentially in a `for...of` loop despite promising concurrent execution. This adds unnecessary latency for independent operations.

**Fix:**
- Use `Promise.allSettled()` with a concurrency limiter (max 5 concurrent) for independent tool calls
- Group tool calls by dependency: file reads can run in parallel, file writes must be sequential per file
- Return aggregated results with success/failure status per tool

**Impact:** Significantly reduces latency for multi-file reads, searches, and independent operations.

---

### 6. Consolidate Duplicate Tool Schema Systems
**Files:** `engine/agentController.js` (line 181-699), `tools/index.js`

**Problem:** Two parallel tool definition systems exist with subtle differences in parameter names, descriptions, and available tools. This creates maintenance burden and inconsistency risk.

**Fix:**
- Keep `TOOLS_SCHEMA` in `agentController.js` as the single source of truth (it's the one actually used by the agent loop)
- Remove the duplicate `TOOL_DEFINITIONS` from `tools/index.js` or generate it from `TOOLS_SCHEMA`
- Ensure `getToolsForRole()` exists in only one place

**Impact:** Eliminates inconsistency risk, simplifies adding new tools.

---

### 7. Add Self-Correction Protocol for Repeated Failures
**File:** `engine/agentController.js` (main loop)

**Problem:** When a tool fails repeatedly (e.g., edit doesn't match 3 times), the model receives generic error strings with no structured guidance on recovery strategies.

**Fix:**
- Track consecutive failures per tool type in the loop
- After 2 consecutive failures of the same tool, inject a system message:
  `"You've had 2 consecutive failures with [tool]. Recovery steps: 1. Read the file to see current state, 2. Try a broader search string, 3. Use write_file as fallback, 4. Ask the user for help."`
- After 4 failures, force a `read_file` of the affected file before allowing another attempt

**Impact:** Prevents the agent from getting stuck in retry loops with the same broken approach.

---

### 8. Fix Non-Functional codebase_search Tool
**File:** `engine/agentController.js` (line 1263)

**Problem:** The `codebase_search` tool is hardcoded to return an error message. It appears in the tool schema, so the model may attempt to use it and always fail.

**Fix:** Either:
- (a) Implement it properly using the existing `searchWorkspaceContent` infrastructure, or
- (b) Remove it from `TOOLS_SCHEMA` to prevent the model from attempting to use it

Option (b) is simpler and safer if Morph API integration isn't planned.

**Impact:** Removes a broken tool from the model's available options.

---

## MEDIUM Priority (Quality of life improvements)

### 9. Improve Command Output Handling
**File:** `tui/components/useChatState.js` (line 854-898)

**Problem:** 
- Command output has no truncation limit — a command producing megabytes of output floods the context
- `shell: true` enables shell injection risk if the model constructs unexpected commands
- No "trust safe commands" option — every command requires manual approval

**Fix:**
- Cap command output at 10KB (configurable)
- Add a "Trust safe commands" option that auto-approves commands from a safe list (npm, git read-only, node, build commands)
- Add error classification to command results: `"compilation error"`, `"missing dependency"`, `"permission denied"` — helps the model apply targeted recovery

**Impact:** Better UX for multi-command workflows, safer execution, faster recovery from errors.

---

### 10. Refresh Smart Context During Long Loops
**File:** `engine/agentController.js` (loop), `engine/context.js`

**Problem:** Smart context is built once at the start and never refreshed. If the agent modifies files during the loop, the context becomes stale — the agent may reference old file contents or miss new files.

**Fix:**
- Call `runtime.buildFreshContext()` every 5 loop iterations
- Cache file contents to avoid re-reading unchanged files
- Replace emoji characters in context strings (`\ud83d\udcc1`, `\ud83d\udcc4`) with plain text labels to save tokens

**Impact:** Agent always works with current file state, better token efficiency.

---

### 11. Remove Duplicate Code and Dead Paths
**Files:** `engine/agentController.js`

**Problem:**
- `question` and `lsp` tools are handled twice — once in `executeToolCall()` (lines 780-803) and again in the main loop (lines 948-973). The main loop version runs; the `executeToolCall` version is dead code.
- `run_command` output handling is duplicated between `executeToolCall()` (line 755) and the main loop (line 1007).
- `PROMPTS` constant (line 173) is never used since `buildSystemPrompt()` loads from disk.
- Two `getToolsForRole()` functions exist (agentController.js and tools/index.js) with different logic.

**Fix:**
- Route all tool execution through `executeToolCall()` consistently, remove duplicate handlers in the main loop
- Extract command output normalization into a shared `normalizeCommandOutput()` function
- Remove the unused `PROMPTS` constant or use it as fallback when file loading fails
- Consolidate to a single `getToolsForRole()` implementation

**Impact:** Cleaner codebase, fewer bugs from divergent code paths, easier maintenance.

---

### 12. Improve Tool Result Summaries
**File:** `engine/agentController.js` (lines 117-154)

**Problem:** Tool result summaries are purely numeric ("Read 45 lines from file.js") with no semantic information about what the file contains.

**Fix:**
- For `read_file`: Include a 1-line content hint from the first few lines: `"Read 45 lines from auth.js (contains login handler and JWT validation)"`
- For error results: Show more of the first meaningful error line (up to 200 chars) instead of truncating at 120
- For `search_files`/`search_content`: Include the most relevant match snippet

**Impact:** Better visibility into what the agent is seeing, easier debugging of agent behavior.

## Expected Outcomes

After implementing all improvements:
- **Fewer hallucinated edits** — edit verification + higher fuzzy threshold prevents wrong replacements
- **Longer successful sessions** — message compaction prevents context overflow
- **Faster execution** — parallel batch reduces latency for multi-file operations
- **Better self-correction** — failure protocols help the agent recover from errors autonomously
- **Clearer model behavior** — resolved prompt contradictions eliminate confused outputs
- **Cleaner codebase** — removed dead code and duplicate systems reduce future bug risk
