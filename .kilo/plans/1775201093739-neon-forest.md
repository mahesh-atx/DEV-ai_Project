# Session-Based Chat Storage with Per-Session Memory

## Overview

Implement session management so that each chat session has its own isolated chat history, activity logs, and memory. Sessions are persisted to `.rootx/sessions/` as JSON files. CLI auto-resumes the last active session instantly. Users can switch, create, rename, and delete sessions from a dedicated Sessions tab in the main menu.

## Architecture

### New Module: `utils/sessionStore.js`
- Session CRUD operations (create, load, list, save, delete, rename)
- Session metadata index with `lastActiveSessionId` for instant resume
- Auto-save on message changes
- Smart title generation (AI-based after 2-3 messages, rule-based fallback)
- Storage format: `.rootx/sessions/<sessionId>.json`
- Index file: `.rootx/sessions/index.json` with lastActiveSessionId

### Session Data Structure
```json
{
  "id": "session-<timestamp>-<random>",
  "title": "Fix CLI parser bug",
  "createdAt": "2026-04-03T13:57:16.000Z",
  "updatedAt": "2026-04-03T14:30:00.000Z",
  "mode": { "value": "agent", "label": "Code" },
  "model": { "key": "kimi", "name": "Kimi K2" },
  "messages": [...],
  "activityLog": [...],
  "summary": { "filesCreated": 0, "filesEdited": 0, ... },
  "customBuildCmd": "",
  "lastCheckpoint": null
}
```

### Index Structure (`.rootx/sessions/index.json`)
```json
{
  "lastActiveSessionId": "session-123",
  "sessions": [
    {
      "id": "session-123",
      "title": "Fix CLI parser bug",
      "createdAt": "...",
      "updatedAt": "...",
      "mode": "Code",
      "model": "Kimi K2",
      "messageCount": 24,
      "lastMessagePreview": "..."
    }
  ]
}
```

## Implementation Steps

### Step 1: Create `utils/sessionStore.js`
- `createSession({ mode, model, projectDir })` — creates a new session with smart title (rule-based from first user message), unique ID
- `saveSession(session, projectDir)` — writes session JSON, updates index, updates `lastActiveSessionId`
- `loadSession(sessionId, projectDir)` — reads and parses session JSON
- `listSessions(projectDir)` — reads index.json, returns sorted by updatedAt DESC
- `deleteSession(sessionId, projectDir)` — removes file and index entry
- `renameSession(sessionId, newTitle, projectDir)` — updates title in session file and index
- `getLastSession(projectDir)` — returns the `lastActiveSessionId` session for instant resume
- `setLastActiveSession(sessionId, projectDir)` — updates `lastActiveSessionId` in index
- `generateSmartTitle(firstUserMessage)` — rule-based fallback: trims to 5-8 words, capitalizes key words
- `getSessionsDir(projectDir)` — returns `.rootx/sessions/` path, creates if missing
- Auto-initialize `.rootx/sessions/` directory on first use

### Step 2: Modify `tui/app.jsx`
- Add new state: `sessionId` (string | null), `sessionTitle` (string)
- Add new view: `'sessionSelect'` for session management
- Add "Sessions" tab to main menu alongside existing options
- When user selects "Sessions" from main menu → show `sessionSelect` view
- When user selects "Start Session" from main menu:
  - If `lastActiveSessionId` exists → auto-resume that session instantly
  - Else → create new session and start chat
- Pass `sessionId`, `sessionTitle`, `onSessionSwitch`, `onNewSession` to `ChatScreen`
- On exit from chat → auto-save current session, update `lastActiveSessionId`, return to main menu

### Step 3: Create `tui/components/SessionSelect.jsx`
- Full session management UI with:
  - List of all sessions sorted by updatedAt DESC
  - Each item shows: **Title** (main), Mode (Code/Chat), relative time, message count
  - Current session highlighted: `● Fix CLI parser bug (active)`
  - Last message preview (1 line, truncated)
- Items: session list, "+ New Session", "< Back"
- On session select → show action submenu: `> Open`, `> Rename`, `> Delete`, `> Back`
- On "Open" → return session ID for loading
- On "Rename" → prompt for new title, call `renameSession`, refresh list
- On "Delete" → confirm, call `deleteSession`, refresh list
- On "New Session" → return null for new session creation

### Step 4: Modify `tui/components/useChatState.js`
- Accept new props: `sessionId` (string | null), `sessionTitle` (string), `onSessionSwitch` (callback), `onNewSession` (callback)
- **On mount**: if `sessionId` provided, load session and restore:
  - `messages` from session.messages
  - `activityLog` from session.activityLog
  - `currentSummaryRef` from session.summary
  - `customBuildCmd` from session.customBuildCmd
  - `lastCheckpointRef` from session.lastCheckpoint
  - `msgIdCounter` to avoid ID collisions (set to max existing ID + 1)
- **On session switch** (triggered by `/switch` or UI):
  - Auto-save current session
  - Clear all in-memory state (messages, activityLog, summary, etc.)
  - Load new session and restore state
  - Update `lastActiveSessionId`
- **On each message change**: auto-save session
- **New slash commands**:
  - `/new` → Create new session instantly (save current, reset state, new session)
  - `/switch` → Trigger session selector (return to sessionSelect view)
  - `/sessions` → List all sessions in chat (show titles, times, message counts)
  - `/delete <id>` → Delete session by ID (with confirmation)
  - `/rename <title>` → Rename current session
- Export `getSessionState()` function that serializes current state for saving

### Step 5: Modify `tui/components/ChatScreen.jsx`
- Pass `sessionId`, `sessionTitle` through to `useChatState`
- Display session title in footer (e.g., `● Fix CLI parser bug`)
- Handle all session slash commands:
  - `/new` → Call `onNewSession`, reset state
  - `/switch` → Set flag to return to sessionSelect on exit
  - `/sessions` → Show session list in chat
  - `/delete <id>` → Delete session, show confirmation
  - `/rename <title>` → Rename current session, update display
- Add visual indicator for current session in header/footer

### Step 6: Update `MainMenu.jsx`
- Add "Sessions" tab: `{ label: 'Sessions', value: 'sessions' }`
- Update "Start Session" label:
  - If `lastActiveSessionId` exists: `Resume: "<title>"` 
  - Else: `Start New Session`
- Show session count: `(N sessions)`

## Key Design Decisions

1. **Auto-resume last session**: `lastActiveSessionId` in index.json enables instant resume on "Start Session"
2. **Smart session titles**: Rule-based fallback from first user message (5-8 words, capitalized). Future: AI-generated after 2-3 messages
3. **Session switching safety**: Auto-save current → clear memory → load new → update lastActiveSessionId
4. **Auto-save**: Session saved after every message exchange. Prevents data loss on crash
5. **Session ID format**: `session-<timestamp>-<random>` for uniqueness and sortability
6. **Index file**: Separate `index.json` avoids reading all session files for listing
7. **Memory scope**: Each session stores its own messages, activity logs, and summary. No cross-session sharing
8. **No auto-cleanup**: Sessions persist indefinitely. Users manage manually

## Edge Cases

- **Session switching mid-work**: Auto-save current → clear in-memory state → load new → update lastActiveSessionId
- **Empty session**: Saved with just init message. Title generated from first user message when available
- **Session with no model config**: Warning shown, prompt to select new model
- **Concurrent sessions**: Only one active at a time in TUI
- **Large sessions**: Full load into memory initially. Future: lazy loading
- **Delete active session**: Prevented with warning. Must switch first
- **Rename while typing**: Title updates immediately in footer and index
