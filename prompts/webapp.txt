ARCHITECTURE PRINCIPLES:
✓ API-first design (REST or GraphQL)
✓ Separation of concerns (MVC pattern)
✓ Proper error handling & validation
✓ Logging and monitoring hooks
✓ Scalable folder structure
✓ Environment-based configuration

==================== STATE MANAGEMENT ====================
Choose based on complexity:
- Simple: React useState/useReducer + Context API
- Medium: Zustand (lightweight, minimal boilerplate)
- Complex: Redux Toolkit (if user specifies or large-scale app)
- Server state: TanStack Query (React Query) for API data caching

Rule: Don't add Redux for a simple app — match complexity to need.

==================== TESTING STRATEGY ====================
Include basic tests for critical paths:
✓ API endpoint tests (status codes, response format)
✓ Auth flow tests (login, register, token validation)
✓ Database operation tests (CRUD correctness)
✓ Input validation tests (edge cases, malicious input)

Tools (defaults):
- Backend: Built-in Node test runner or Jest
- Frontend: Vitest + React Testing Library
- E2E: Playwright or Cypress (only if user requests)

Minimum: Include at least a test file for critical API routes.

==================== DEPLOYMENT RULES ====================
✓ Include production build scripts in package.json
✓ Separate dev/prod environment configs (.env.development, .env.production)
✓ Add health check endpoint: GET /api/health → { status: "ok" }
✓ Include Dockerfile (optional, if user mentions deployment)
✓ Database migrations or init script for fresh setup
✓ README with clear setup, run, and deploy instructions

==================== PERFORMANCE ====================
✓ Use pagination for list endpoints (limit, offset or cursor)
✓ Cache frequent read queries (in-memory or Redis)
✓ Use database indexes on frequently queried columns
✓ Lazy load frontend routes and heavy components
✓ Debounce search inputs and expensive UI operations
✓ Compress API responses (gzip via Express compression middleware)
✓ Use connection pooling for database (if not SQLite)
